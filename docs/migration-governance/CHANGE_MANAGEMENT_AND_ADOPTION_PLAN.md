# Change Management and Adoption Plan

## 1. Purpose

This document defines the user adoption strategy for a BI platform migration. It covers user segmentation, communication, training, office hours, legacy report decommissioning, adoption monitoring, support model, and feedback loops.

Change management in a BI migration is not supplementary—it is part of the release. A technically successful migration that users do not adopt, cannot navigate, or do not trust produces no business value. This document treats adoption as a measurable outcome, not an implied result of technical delivery.

---

## 2. Business Use Case

When users have been working with the same reporting environment for years, their workflows are built around it. They know which report to open, which filters to apply, what the numbers should look like. A platform migration changes all of that at once. Without deliberate change management, the result is increased support volume, decreased confidence in data, users who revert to workarounds (spreadsheets, offline exports), and a perception that the new platform is worse than the old one—regardless of its technical quality.

A structured adoption plan reduces these risks by:

- Preparing users before cutover, not after
- Segmenting users by role and data sophistication so training is relevant to what they actually do
- Creating a feedback channel so user concerns inform platform improvements
- Monitoring adoption so the platform team knows which reports are being used and which are not
- Decommissioning the legacy platform on a defined schedule so the parallel period does not become permanent

| Role | Adoption Responsibility |
|---|---|

## 5. Operating Model

### Communication Model

Communication follows a sequenced timeline relative to the cohort release date.

| Timing | Audience | Channel | Content |
|---|---|---|---|
| 4 weeks before | Department managers, dataset owners | Email + meeting | Migration overview, timeline, what changes for their team |
| 3 weeks before | All affected users | Email | Migration announcement; what Power BI means for them; what stays the same |
| 2 weeks before | UAT participants | Direct invitation | UAT period invitation; test access instructions |
| 1 week before | All affected users | Email | Go-live date reminder; training session schedule; support contact |
| Day of cutover | All affected users | Email | Go-live confirmation; links to Power BI reports; support contact; OBIEE still available |
| Day 3 post-cutover | All affected users | Email | Check-in; feedback link; office hours schedule |
| Day 14 post-cutover | All affected users | Email | Hypercare close; ongoing support model; OBIEE decommission date |
| Day 28 post-cutover | All affected users | Email | Final OBIEE reminder; decommission in 2 days |
| Day 30 | All affected users | Email | OBIEE decommission confirmed; Power BI is the system of record |

**Executive sponsor communication:** The executive-level announcement (3 weeks before) carries the message that the platform change is a strategic direction, not an IT project. Executive visibility at the announcement stage reduces resistance and signals to department managers that adoption is expected.

### User Segmentation

Users are segmented into three tiers based on how they interact with reports:

**Tier 1: Report consumers (view-only)**
Navigate to existing reports, apply standard filters, read output. These users need: a map from old report names to new report names, a navigation guide for Power BI Service, and a support contact.

**Tier 2: Power users (create and customize)**
Modify reports, create personal bookmarks, use drill-down and cross-filtering, build ad hoc views. These users need: all Tier 1 materials, plus a training session on Power BI interactive features, bookmarks, and personal views.

**Tier 3: Report builders (contribute to the platform)**
Build new reports against certified datasets, write DAX measures, manage workspace content. These users need: all Tier 2 materials, plus dataset documentation review (data dictionary, RLS mapping, certification criteria), and onboarding to the intake and change approval process.

Training content and depth varies by tier. A single generic training session serves no tier well.

## 6. Architecture

### Training Architecture

Training is role-based and timed to arrive before users need it—not after go-live when they are already frustrated.

| Training Type | Audience | Format | Timing |
|---|---|---|---|
| Executive overview | Department managers, dataset owners | 30-min briefing | 4 weeks before cutover |
| Report consumer orientation | Tier 1 users | 45-min live or recorded session | 1–2 weeks before cutover |
| Power user workshop | Tier 2 users | 60-min interactive session with sample data | 1–2 weeks before cutover |
| Report builder onboarding | Tier 3 users | 90-min session covering dataset docs, intake process, change approval | 1 week before cutover |
| Office hours | All tiers | Drop-in (1 hour, 3x per week) | First 3 weeks post-cutover |

**Quick-start guide:** A single-page reference card distributed at go-live showing:
- How to access Power BI Service
- How to find the reports you used to use in OBIEE (naming map)
- How to apply filters and use slicers
- How to get help

### Report Naming Map

One of the highest-friction elements of a platform migration is users not being able to find their reports. The naming map addresses this by providing a direct translation from every OBIEE report name to its Power BI equivalent.

The naming map is generated from `config/migration_inventory.yaml` using `scripts/generate_mapping_report.py`. It is distributed as a reference document at go-live and embedded in the quick-start guide.

Example mapping format (from the inventory YAML):

| OBIEE Report Name | Subject Area | Power BI Report Name | Power BI Dataset | Notes |
|---|---|---|---|---|
| GL Journal Detail | Finance - GL | GL Journal Detail | Finance | Same name; same columns |
| AP Vendor Summary | Finance - AP | AP Vendor Summary | Finance | Count variance documented: reversed entries excluded |
| Budget vs Actual | Finance - Budget | Budget vs Actual | Finance | Reforecast behavior documented in data dictionary |

### Office Hours Model

Office hours provide a structured, low-barrier channel for users to get help without creating formal support tickets. Office hours are particularly effective in the first 2–3 weeks when users are discovering the new platform in their daily workflow.

**Office hours format:**
- Duration: 60 minutes
- Frequency: 3 times per week for the first 3 weeks post-cutover; 1 time per week for weeks 4–8
- Facilitator: 1 platform team member with dataset knowledge and Power BI proficiency
- Format: Open drop-in; no registration required
- Channel: Virtual (video call with screen sharing)

**Office hours governance:**
- Issues that require platform changes are logged in the Data Quality Issue Board or Platform Operations Board, not resolved ad hoc during office hours
- Patterns of recurring questions inform FAQ updates and training content revision
- Office hours attendance is tracked; low attendance does not indicate low need—it may indicate users are not aware of the option or the time slots don't work

## 7. Controls

### Adoption Monitoring

Adoption is measured to confirm that the platform is being used and to identify reports or user groups where adoption is lagging.

**Usage monitoring (Power BI Service)**
Power BI Service provides usage metrics at the workspace and report level:
- Views per report per day
- Unique viewers per report per week
- Top reports by view count

These metrics confirm whether users are accessing Power BI reports. They do not confirm whether users are getting the information they need.

**OBIEE access monitoring (during parallel run)**
During the 30-day parallel run, OBIEE access logs identify users who continue using OBIEE rather than transitioning to Power BI. High OBIEE usage for a specific report after cutover is a signal to investigate: is the Power BI version missing something, or does the user need additional support?

**Feedback collection**
A structured feedback form is distributed at day 3 post-cutover and again at day 14. The form captures:
- Which report the user was trying to use
- Whether they were able to accomplish their task
- If not, what was missing or wrong
- Open-ended: what would make the platform more useful for their work

Feedback is reviewed weekly by the platform team and dataset owners. High-frequency themes are prioritized for resolution.

### Legacy Report Decommissioning

Decommissioning OBIEE on a defined schedule prevents the parallel period from becoming permanent, which would undermine adoption, create data governance confusion, and sustain the maintenance burden of the legacy platform.

**Decommissioning conditions:**
- 30-day parallel run period complete
- No Critical or unresolved High issues open for the cohort
- Platform Lead confirms decommission authorization

**Decommissioning sequence:**
1. Day 28: Final reminder to all users that OBIEE will be decommissioned in 2 days
2. Day 30: OBIEE access removed for reports in the completed cohort
3. Decommission confirmed in Migration Status Board (OBIEE Decommission Date field updated)
4. Any OBIEE bookmarks or direct links in user communications are updated

**If decommissioning is delayed:** A delay is documented with a specific reason and a new target date. Indefinite delay is not permitted without platform lead approval and a documented exception.

## 8. Validation Workflow

Adoption plan effectiveness is validated through observable metrics, not assumption.

**Pre-cutover validation:**
- Training sessions completed and attendance logged
- Quick-start guide distributed and receipt confirmed (email read receipts or confirmation response)
- Support channel (office hours, helpdesk contact) communicated and visible in all materials

**Post-cutover validation (week 1):**
- Power BI usage metrics reviewed: are users accessing reports?
- OBIEE access logs reviewed: are users still primarily using OBIEE for migrated reports?
- Office hours attendance reviewed: are questions clustering around specific reports or workflows?
- Helpdesk ticket volume by category reviewed: are tickets about navigation, data accuracy, or access?

**Post-cutover validation (week 2–4):**
- Feedback form responses reviewed and themes documented
- Adoption rate by user tier compared against targets
- Reports with zero or near-zero usage identified and investigated (may indicate accessibility issue or decommission candidate in the next rationalization cycle)

```
        │
        ▼

Ongoing: Usage monitoring + feedback loops + quarterly review
```

## 10. Operational Metrics

These are the metrics the platform team and dataset owners monitor to evaluate adoption progress. No adoption outcomes are claimed for this framework. These are the metrics to track, not results to assert.

| Metric | Description | Measurement Source |
|---|---|---|
| Weekly active users | Count of unique users accessing Power BI reports per week | Power BI Service usage metrics |
| Report activation rate | % of migrated reports with at least 1 view in first 30 days | Power BI Service usage metrics |
| OBIEE parallel usage rate | % of migrated reports still accessed in OBIEE during parallel run | OBIEE access logs |
| Support ticket volume | Count of user-reported issues per week post-cutover | Helpdesk/Platform Operations Board |
| Feedback sentiment | Qualitative theme summary from feedback forms | Feedback form responses |
| Office hours attendance | Headcount per session | Office hours attendance log |
| Training completion | % of Tier 1/2/3 users completing appropriate training | Training attendance records |
| Decommission on schedule | Boolean: was OBIEE decommissioned within 30 days of cutover? | Migration Status Board |

## 11. Workplace Application

This adoption model applies to any enterprise platform migration where users must change established workflows. The specific platform (Power BI, Tableau, Looker, any modern BI tool) is substitutable. The adoption structure is not.

**What transfers directly:**

- The user segmentation model (consumer / power user / builder) applies to any reporting platform transition
- The communication timeline relative to cutover date applies regardless of platform
- The report naming map approach applies to any migration where report names change
- The office hours model applies to any organization-facing platform change
- The decommissioning governance model applies to any legacy system sunset

**What must be adapted:**

- Training content is platform-specific (Power BI features, navigation, interactive capabilities)
- Usage metrics depend on what the target platform exposes (Power BI Service has built-in usage reporting; other platforms may require custom instrumentation)
- Communication channels depend on organizational norms (email, Teams, Slack, intranet)

---

## 12. Limitations

- Adoption metrics require Power BI Service access and usage reporting to be enabled. This repo does not include Power BI Service configuration; usage monitoring is a post-deployment activity.
- OBIEE access log monitoring during the parallel run requires OBIEE administrator access. This is not implemented in this repo.
- Training materials (slide decks, recorded sessions, quick-start guides) are not artifacts in this repo. This document defines the training architecture and content requirements; the materials themselves are produced during the migration.
- The feedback form referenced is a design pattern, not a deployed survey. The form content, distribution mechanism, and response collection are defined by the organization's tooling.
- No adoption outcome metrics are available from this repo's migration engagement. Usage data, training attendance, and feedback responses are not included in this public repository.

---

## 13. What This Does Not Claim

- This document does not claim specific adoption rates, training completion percentages, or user satisfaction scores. No such data is available from this engagement in the public repo.
- This framework does not claim that the communications and training materials described here were executed and confirmed effective in a specific organization. The plan is a designed framework, not a deployment record.
- This framework does not claim that OBIEE was decommissioned on day 30 for all cohorts. The migration playbook defines the target; actual decommission timelines depend on organizational factors outside the scope of this repo.
- This framework does not cover organizational change management beyond the BI platform scope (workforce changes, org redesign, process reengineering). Those are separate disciplines.

## 14. Extension Path

- **Automated usage reporting:** Build a script that queries Power BI REST API usage metrics and generates a weekly adoption dashboard showing views, active users, and zero-usage reports by cohort
- **Feedback analysis integration:** Route feedback form responses into the Data Quality Issue Board (via Monday.com GraphQL) so user-reported issues automatically create tracked items with severity classification
- **Training content repository:** Add a `training/` directory to the repo with quick-start guide templates, report naming map format, and sample UAT test scenario templates
- **Report rationalization trigger:** Extend the adoption monitoring model to flag reports with zero usage after 90 days post-cutover as candidates for decommission, feeding back into the intake and prioritization process
- **Cohort retrospective template:** Add a structured retrospective template to capture what worked and what to improve for each cohort release, creating organizational learning that accelerates subsequent releases

## 15. Interview Talking Points

**On why change management belongs in the technical framework:**
> "Change management isn't a soft track that runs in parallel with the technical migration. It's a dependency. If users aren't trained and the platform isn't communicated properly, you'll see support volume spike post-cutover and users reverting to spreadsheets. That undermines all the technical work. The adoption plan is part of the release definition."

**On user segmentation:**
> "A single training session for all 500 users is almost useless. Someone who only ever opens a pre-built report and reads the number needs completely different preparation than someone who builds reports. The consumer needs to know how to find their report in Power BI and what changed. The builder needs to understand the data dictionary and how to get through the intake process. One session can't serve both."

**On the report naming map:**
> "The single most friction-generating moment in a platform migration is when a user opens Power BI for the first time and can't find their report. The naming map eliminates that friction. It's a simple table—OBIEE name, Power BI name, dataset—distributed on day one. Without it, you'll spend the first week of support just answering 'where is my report.'"

**On decommissioning governance:**
> "The parallel run period has to have a defined end. If you leave OBIEE running indefinitely, users will never fully commit to Power BI. Every time they find a discrepancy, they'll check OBIEE and trust it over Power BI by default, even if Power BI is correct. The 30-day window gives users enough time to develop confidence, and the defined decommission date forces the commitment. The decommission delay governance prevents the parallel period from becoming permanent through organizational inertia."

**On feedback loops:**
> "Adoption monitoring is not the same as usage monitoring. Usage tells you whether people opened the report. Feedback tells you whether they got what they needed. Both matter. A report with high views and consistent feedback saying 'I can't find my filter' is not adopted—it's tolerated. The feedback loop is what closes the gap between 'technically deployed' and 'actually useful.'"
